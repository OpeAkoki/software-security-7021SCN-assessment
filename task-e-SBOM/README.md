# Task E — Compliance and Supply Chain

## What this task covers

This task has two parts. First, generating a Software Bill of Materials (SBOM) for the webapp automatically via a CI pipeline and scanning it for known vulnerabilities in its dependencies. Second, mapping all the findings from the assessment back to the NIST SSDF (SP 800-218) to show which secure development practices were missing and what should have been done differently.

## SBOM Generation

The SBOM is generated automatically by the GitHub Actions pipeline at `.github/workflows/generate-sbom.yml`, which triggers on every push to main. The pipeline uses the official Anchore `sbom-action` to run Syft against the webapp directory in a clean, ephemeral Ubuntu runner. The output format is CycloneDX JSON, which is one of the two main SBOM standards alongside SPDX.

Generating the SBOM in CI rather than locally means the build environment is never contaminated by a developer's local machine, satisfying SSDF PS.1. The SBOM artefact is uploaded by the pipeline and a copy is stored in `sbom/sbom.cdx.json` for direct inspection.

## Vulnerability Scan

The SBOM was scanned using **Grype** via Docker to check all listed components against known CVE and GHSA databases.

Command used:

```bash
docker run --rm -v $(pwd)/task-e-SBOM/sbom:/sbom anchore/grype sbom:/sbom/sbom.cdx.json
```

Two vulnerabilities were found, both in `pip` version 24.0. Neither affects the running webapp directly but both should be remediated by upgrading pip to version 26.0 or later.

## SSDF Mapping

`ssdf-mapping.md` maps each finding from Tasks A through D to the relevant NIST SSDF SP 800-218 practice. It covers secure coding (PW.5), input validation (PW.6), code review (PW.7), testing (PW.8), and vulnerability management (RV.1 through RV.3). A second section maps each portfolio artifact to the SSDF practice it actively satisfies.
